Database Management Project

Project Overview

In this project, student teams will design and build a database-driven solution that addresses a real-world problem with potential to evolve into a usable product or service. Teams will identify their own problem domain and develop a system that demonstrates both strong technical design and meaningful business value.

Project Objectives

The project should:

  • Solve a clearly defined real-world problem
  • Demonstrate end-to-end database design and implementation
  • Deliver actionable insights through SQL and analytics
  • Include a basic user-facing interface enhanced with AI assistance
  • Show potential to scale into a real product

Core Requirements

1. Problem Definition & Value Proposition

  • Clearly define the problem being solved
  • Identify target users/stakeholders
  • Explain why the solution is valuable and how it improves current processes

2. Conceptual Design (ERD)

  • Develop a well-structured Entity-Relationship Diagram (ERD)
  • Include entities, attributes, keys, and relationships
  • Demonstrate appropriate handling of:
  • One-to-many / many-to-many relationships
  • Weak entities or associative entities (if applicable)

3. Logical Design & Database Implementation

  • Convert ERD into a normalized relational schema (up to 3NF)
  • Implement the database in PostgreSQL (or equivalent)
  • Create tables with appropriate:
  • Primary and foreign keys
  • Constraints (NOT NULL, UNIQUE, etc.)

4. SQL Query Development

Develop a set of SQL queries that support both operations and analytics:

  • Basic queries (SELECT, WHERE, ORDER BY)
  • Multi-table joins (at least 3–4 tables)
  • Aggregations (GROUP BY, HAVING)
  • Business-driven queries answering real stakeholder questions

5. Analytics & Insights

  • Define key metrics/KPIs relevant to the problem
  • Use SQL to generate insights (e.g., trends, rankings, segmentation)
  • Present results in a clear, business-friendly format

6. AI-Assisted Interface (Lightweight Front-End)

  • Build a simple interface (e.g., Streamlit, web app, or dashboard)
  • Integrate AI assistance (e.g., natural language query, chatbot, or insights explanation)
  • Enable non-technical users to interact with the database

Deliverables

  • ERD diagram
  • Database schema and SQL scripts
  • Set of business-driven queries
  • Working interface/demo
  • Peer Feedback
  • Final presentation (problem → solution → impact → demo)

Evaluation Criteria

  • Technical Design (ERD, normalization, schema quality)
  • SQL proficiency and query complexity
  • Business relevance and clarity of insights
  • Usability and creativity of interface (AI integration)
  • Product potential (real-world applicability, scalability)

Key Expectation

This is not just a database project.
Teams are expected to think like product builders and design something that a real user could adopt to solve a real problem.

Idea

1. The "Product" Concept: EthiTrack

The Problem: Consumers want to support ethical brands or buy local produce but find it hard to track price trends, seasonal availability, and the "social impact score" of their purchases. The Solution: A database-driven platform that tracks product pricing and trends while highlighting items from NPO-partnered vendors or sustainable sources.

2. Conceptual Design: The "Nouns" (Entities)

To satisfy the project requirements, your database needs to be more than a simple list. You’ll need these core entities:

  • Products: Name, category, base price, sustainability rating.
  • Vendors: Name, location, type (NPO-partnered, Local, Corporate), and their "Impact Score."
  • Price_Logs: Historical data points (Price, Date, Vendor_ID) to track the "Trend."
  • Categories: For segmentation (e.g., Organic Food, Ethical Apparel, Fair-Trade Coffee).
  • Consumers/Users: Profiles to track "Watchlists" or "Saved Trends."

3. Your Immediate First Step: Define the "Value Proposition"

Before drawing the ERD, write down your Core Business Questions. These will prove the "Business Value" requirement of your project.

Examples of questions your database will answer:

  • "Which NPO-affiliated vendor offers the lowest price for Fair-Trade Coffee in the current month?"
  • "What is the 6-month price trend for organic kale, and when is it typically cheapest?"
  • "Which products have a 'Sustainability Rating' above 80 but are currently priced below the market average?"

4. Why this works for your Professor

  • Non-Profit Benefit: You can argue that this platform promotes NPO-backed businesses and helps low-income consumers find the best deals on high-quality goods.
  • Technical Depth: Tracking "Trends" requires a One-to-Many relationship between a Product and its Price_Logs. This allows for the complex SQL joins and aggregations (like AVG() or MAX()) required in the rubric.
  • AI Opportunity: Your "AI-Assisted Interface" can take a user's natural language query (e.g., "Find me a cheap ethical winter coat") and use the database to explain the price trend.

1. The ERD Entities & Attributes

Core Entities

  • Product: The "What."
  • product_id (PK), product_name, brand, base_description, category_id (FK).
  • Vendor: The "Where."
  • vendor_id (PK), vendor_name, is_npo_partner (Boolean), location_type (Local/Online), sustainability_score.
  • Category: To organize data.
  • category_id (PK), category_name (e.g., Fair Trade Coffee, Organic Textiles).

Relational/Trend Entities

  • Price_Listing: This connects Products to Vendors (Many-to-Many).
  • listing_id (PK), product_id (FK), vendor_id (FK), current_price, stock_status.
  • Price_History: This is the "secret sauce" for your Trends.
  • history_id (PK), listing_id (FK), recorded_price, date_timestamp.
  • Note: This allows you to say "Price was $10 in Jan, but $12 in Feb."

User Interaction

  • User:
  • user_id (PK), username, email, impact_preference (e.g., "Prefers NPO over Price").
  • Watchlist:
  • watchlist_id (PK), user_id (FK), product_id (FK), target_price.

3. Why this design satisfies your Professor

  1. Complexity: The Price_History table ensures you aren't just looking at a static list. You’ll have to use AVG() and GROUP BY in SQL to show the "Trend."
  2. NPO Integration: The is_npo_partner flag in the Vendor table allows you to generate "Actionable Insights" (Requirement #5), like: "Show me the price trend for products exclusively sold by NPO-partnered vendors."
  3. 3rd Normal Form (3NF): By separating Price_History from Product, you avoid data redundancy.

4. Next Steps for your Team

  1. Finalize Attributes: Decide exactly what "Trend" data you want. Do you want to track "Discount Percentage" or just "Price"?
  2. Draw the Diagram: Use a tool like Lucidchart, draw.io, or even MySQL Workbench to create the formal ERD with crow's foot notation.
  3. Data Sourcing: Start thinking about where you'll get data. You can find "Open Food Facts" datasets or "Non-Profit Directories" to populate your tables with realistic information.
